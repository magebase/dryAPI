# GenFix Next.js + TinaCMS + OpenNext (Cloudflare)

Original industrial website implementation inspired by the provided reference structure, with all major content editable via TinaCMS and media stored on Cloudflare R2.

## Stack

- Next.js App Router + TypeScript
- Tailwind CSS + custom design system tokens
- TinaCMS (JSON collections, WYSIWYG editing UI)
- Cloudflare R2 media upload/list/delete API integration
- OpenNext for Cloudflare (`@opennextjs/cloudflare`)
- Resend + React Email for contact inquiries
- Zod schema validation
- Vitest

## Local Development

```bash
npm install
npm run dev
```

`npm run dev` starts Next.js and Tina's local GraphQL service together.
If you need plain Next.js without Tina, use:

```bash
npm run dev:next
```

Tina local authoring:

```bash
npm run dev:tina
```

## Editable Content Model

All visible site data lives in JSON under `content/`.

- `content/site/site-config.json`
  - Header links, phone CTA, quote CTA
  - Footer columns, social links, legal links
- `content/site/home.json`
  - Hero copy/buttons/image
  - Spotlight cards, capability cards
  - Project/resource showcase cards
- `content/pages/*.json`
  - One document per route page (hero, sections, cards, optional contact panel)
- `content/blog/*.json`
  - One document per blog post (meta, author, tags, cover image, article sections)

Schemas and loaders:

- `src/lib/site-content-schema.ts`
- `src/lib/site-content-loader.ts`

## TinaCMS + R2 Workflow

1. Editor opens Tina UI.
2. Edits JSON content (text, links, cards, layout content).
3. Uploads images through Tina media store.
4. Upload API writes media to Cloudflare R2:
   - `POST /api/media/upload`
   - `GET /api/media/list`
   - `POST /api/media/delete`
5. URLs are stored in content JSON.
6. Content commits trigger a rebuild/deploy.

Quote submissions are routed to the Tina-managed `contact.quoteEmail` value in:

- `content/site/site-config.json`

This address is validated to the `@genfix.com` domain in the runtime schema.

## OpenNext + Cloudflare Hosting Pattern

Cloudflare worker config:

- `open-next.config.ts`
- `wrangler.jsonc`

Build and deploy commands:

```bash
npm run cf:build
npm run cf:deploy
```

Drizzle + D1 commands:

```bash
pnpm db:generate
pnpm db:migrate:local
pnpm db:migrate:remote
```

`db:migrate:*` commands require `CF_D1_DATABASE_NAME` to be set in your shell.

### D1 Binding

`wrangler.jsonc` includes a `QUOTE_DB` binding for quote request persistence.
Set the real D1 database ID before deploy:

```json
"d1_databases": [
  {
    "binding": "QUOTE_DB",
    "database_name": "genfix-db",
    "database_id": "<your-d1-database-id>",
    "migrations_dir": "drizzle/migrations"
  }
]
```

Local production preview:

```bash
npm run cf:build
npx wrangler dev
```

## Environment Variables

Copy `.env.example` to `.env.local` and set values.

Required for Tina + R2 media:

- `R2_ACCOUNT_ID`
- `R2_BUCKET`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_PUBLIC_URL`

Required for Tina Git mode:

- `NEXT_PUBLIC_TINA_CLIENT_ID`
- `TINA_TOKEN`

Optional Tina endpoint override:

- `NEXT_PUBLIC_TINA_CONTENT_API_URL` (defaults to `/api/tina/gql`)

Optional: Better Auth as Tina auth provider (`bring-your-own` auth mode):

- `TINA_AUTH_PROVIDER=better-auth`
- `BETTER_AUTH_SECRET` (required in production)
- `BETTER_AUTH_URL` (recommended, e.g. `https://your-domain.com`)
- `TINA_AUTH_TOKEN_SECRET` (recommended separate secret for Tina editor JWTs)
- `TINA_BETTER_AUTH_PROVIDER` (optional, defaults to `google`)
- `TINA_ALLOWED_GOOGLE_EMAILS` and/or `TINA_ALLOWED_GOOGLE_DOMAINS` (editor allowlist)
- `GOOGLE_CLIENT_ID` + `GOOGLE_CLIENT_SECRET` (if using Google social login)
- `GITHUB_CLIENT_ID` + `GITHUB_CLIENT_SECRET` (if using GitHub social login)

When `TINA_AUTH_PROVIDER=better-auth`, the middleware bypasses Cloudflare checks for Tina paths and uses Better Auth + signed Tina editor tokens for `/api/tina/gql` authorization.

Required for Cloudflare Zero Trust protection of TinaCMS routes:

- `CLOUDFLARE_ACCESS_TEAM_DOMAIN` (example: `your-team.cloudflareaccess.com`)
- `CLOUDFLARE_ACCESS_AUD` (Cloudflare Access app audience; comma-separated if multiple)
- `TINA_ALLOWED_GOOGLE_EMAILS` (comma-separated allowlist, e.g. `editor1@gmail.com,editor2@company.com`)
- `TINA_ALLOWED_GOOGLE_DOMAINS` (optional comma-separated domain allowlist)

Required for contact emails:

- `RESEND_API_KEY`
- `RESEND_FROM_EMAIL`
- `RESEND_TO_EMAIL`

Required for D1 migration commands:

- `CF_D1_DATABASE_NAME`

For Cloudflare deploy secrets, prefer Wrangler secrets for sensitive values:

```bash
wrangler secret put R2_ACCESS_KEY_ID
wrangler secret put R2_SECRET_ACCESS_KEY
wrangler secret put TINA_TOKEN
wrangler secret put CLOUDFLARE_ACCESS_TEAM_DOMAIN
wrangler secret put CLOUDFLARE_ACCESS_AUD
wrangler secret put TINA_ALLOWED_GOOGLE_EMAILS
wrangler secret put RESEND_API_KEY
wrangler secret put BETTER_AUTH_SECRET
wrangler secret put TINA_AUTH_TOKEN_SECRET
```

## Cloudflare Zero Trust for TinaCMS

Use this section when `TINA_AUTH_PROVIDER` is not set to `better-auth`.

Protected paths:

- `/admin/*`
- `/api/cms/*`
- `/api/media/*`
- `/api/tina/*`
- `/api/verify-zjwt`

How it works:

1. Cloudflare Access handles Google sign-in and policy enforcement.
2. Origin verifies `Cf-Access-Jwt-Assertion` against Cloudflare Access JWKS and audience.
3. Origin checks the authenticated email against your allowlist.

Quick verification:

```bash
curl -i https://<your-domain>/api/verify-zjwt
```

Expected responses:

- `200` for an authenticated, allowlisted account.
- `401` for missing/invalid/expired token.
- `403` for authenticated but non-allowlisted account.

## CI/CD: GitHub Actions -> Cloudflare

Workflow file: `.github/workflows/cloudflare-deploy.yml`

What it does on every commit push:

1. Installs dependencies.
2. Syncs `SYNC_SECRET_*` GitHub Actions env values to Cloudflare Worker secrets.
3. Applies D1 migrations remotely.
4. Builds OpenNext output.
5. Deploys to Cloudflare Workers.

Secret sync script:

- `scripts/sync-github-secrets.sh`

GitHub repository secrets expected by the workflow:

- `CLOUDFLARE_API_TOKEN`
- `CLOUDFLARE_ACCOUNT_ID`
- `CLOUDFLARE_WORKER_NAME`
- `CF_D1_DATABASE_NAME`
- `NEXT_PUBLIC_TINA_CLIENT_ID`
- `TINA_TOKEN`
- `CLOUDFLARE_ACCESS_TEAM_DOMAIN`
- `CLOUDFLARE_ACCESS_AUD`
- `TINA_ALLOWED_GOOGLE_EMAILS`
- `TINA_ALLOWED_GOOGLE_DOMAINS` (optional)
- `R2_ACCOUNT_ID`
- `R2_BUCKET`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_PUBLIC_URL`
- `R2_ENDPOINT`
- `RESEND_API_KEY`
- `RESEND_FROM_EMAIL`
- `RESEND_TO_EMAIL`
- `UPSTASH_REDIS_REST_URL`
- `UPSTASH_REDIS_REST_TOKEN`

Any additional GitHub secret can be synced by adding it to workflow `env` with the `SYNC_SECRET_` prefix.
